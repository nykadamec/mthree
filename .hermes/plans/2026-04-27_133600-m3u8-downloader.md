# M3U8 Video Downloader — Plan

## Goal

A simple React web app that lets the user paste an `.m3u8` URL, downloads and converts it to `.mp4` using FFmpeg on the backend, and exposes a download link for the resulting file.

## Tech Stack

- **Frontend**: React (Vite), plain CSS, minimal UI
- **Backend**: Node.js + Express
- **FFmpeg**: called via `child_process.exec` / `spawn` on the server
- **File delivery**: served statically from the server's `output/` directory

---

## What the app does (user flow)

1. User opens the page, sees a single input field: "Paste your .m3u8 URL here"
2. User pastes the URL and clicks "Download"
3. Frontend POSTs the URL to the backend
4. Backend runs `ffmpeg -i "<url>" -c copy -bsf:a aac_adtstoasc -movflags +faststart <output>.mp4`
5. While FFmpeg runs, the UI shows a progress indicator (polling `/status/<jobId>`)
6. When done, a "Download .mp4" link appears pointing to the static file

---

## API Design

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/download` | Body: `{ url: string }`. Returns `{ jobId: string }` |
| GET | `/status/<jobId>` | Returns `{ state: "running" \| "done" \| "error", message?: string, downloadUrl?: string }` |
| GET | `/downloads/<filename.mp4>` | Serves the finished file |

## Step-by-Step Plan

### Phase 1 — Project scaffolding

1. Create root `package.json` with `concurrently`, scripts for dev
2. Create `server/package.json` with `express`, `cors`, `uuid`
3. Create `server/index.js`:
   - In-memory job map (`jobId → { state, message, outputPath }`)
   - POST `/api/download` — validate URL, generate jobId, `spawn` FFmpeg asynchronously
   - GET `/status/:jobId` — return job state
   - GET `/downloads/:file` — `res.sendFile` from `output/`
   - Static `output/` dir creation on startup
4. Create `client/` with Vite + React (`npm create vite@latest client -- --template react`)
5. Add `"proxy": "http://localhost:3001"` to `client/vite.config.js`

### Phase 2 — Frontend UI

6. `App.jsx`:
   - State: `url` (input), `jobId`, `status`, `downloadUrl`, `error`
   - Form: POSTs to `/api/download`, starts polling `/status/<jobId>` every 2 s
   - Render: input + button, status text, download link when ready
7. `App.css`: minimal dark styling, clean centered card, warm orange accent `#ff6b35`

### Phase 3 — FFmpeg integration & progress parsing

8. In `server/index.js`, FFmpeg spawn with progress parsing:
   ```js
   ffmpeg -i "<url>" -c copy -bsf:a aac_adtstoasc -movflags +faststart output/<jobId>.mp4
   ```
   - FFmpeg prints to stderr in format: `frame= 1234 fps= 30 q=-0.0 size=    5678kB time=00:01:23.45 bitrate= 456.7kbits/s speed=1.23x`
   - Parse `time=` from stderr → calculate total duration estimate → emit progress to job store
   - On exit code 0 → set job state to `done`, set `downloadUrl`
   - On non-zero → set state to `error` with stderr snippet
9. Progress struct stored in job:
   ```js
   { state, progress: 0-100, message, downloadUrl }
   ```
10. Cleanup old files (optional): cron or on-new-job delete files older than 24 h

### Phase 4 — Polish & validation

10. Validate that the m3u8 URL contains `.m3u8` before spawning FFmpeg
11. Handle CORS (server uses `cors` middleware)
12. Verify both servers start cleanly with `npm run dev`
13. Test full flow: paste real m3u8 URL → get .mp4 → download link appears

## Risks & Tradeoffs

- **FFmpeg must be installed on the server** — add a startup check that errors clearly if `ffmpeg` is not in PATH
- **Large files** — no streaming to client; file lands on disk first. Fine for moderate-sized videos; for very large files consider range requests or object storage
- **No authentication** — anyone who can reach the server can trigger downloads. No scope for this version but worth noting
- **Disk space** — no auto-cleanup in v1; manually clear `output/` or add cleanup
- **m3u8 may be encrypted (AES-128)** — this plan assumes a plain HLS stream. Encrypted streams need `-hls_key_info_file` flag; can be extended later
